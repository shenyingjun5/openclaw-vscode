# 技术决策记录

## ADR-001: ChatController 共享架构

**日期**: 2026-02-08

**状态**: 已采纳

**背景**:
在 v0.2.5 之前，侧边栏（`ChatProvider`）和独立面板（`ChatPanel`）各自实现聊天逻辑，导致代码重复、维护困难、功能不一致。

**决策**:
将聊天核心逻辑抽取到 `ChatController` 类，供侧边栏和独立面板共享。

**理由**:
1. 避免代码重复（减少约 40% 代码量）
2. 统一业务逻辑，减少维护成本
3. 便于功能扩展和 bug 修复
4. 提高代码可测试性

**影响**:
- 重构了 `ChatProvider` 和 `ChatPanel`，移除重复代码
- 新增 `ChatController` 类（约 800 行）
- 新功能只需在一处实现
- 状态管理更加清晰

**替代方案**:
1. 继续维护两份代码（被拒绝：维护成本高）
2. 使用继承（被拒绝：不够灵活）
3. 使用组合（已采纳：灵活且易于测试）

## ADR-002: 聊天状态机设计

**日期**: 2026-02-08

**状态**: 已采纳

**背景**:
需要追踪聊天状态（发送中、等待回复、空闲），以便正确显示 UI 状态（thinking indicator、发送按钮）和管理消息队列。

**决策**:
使用 `isSending` 和 `chatRunId` 两个状态变量追踪聊天状态。

**状态定义**:
- `isSending = true`：正在发送 RPC 请求
- `chatRunId != null`：等待 AI 回复
- `isBusy() = isSending || !!chatRunId`：忙碌状态

**理由**:
1. 对齐 webchat 的状态机设计
2. 避免 chat 事件先于 RPC 响应到达时被跳过
3. 支持消息队列（忙碌时排队）
4. 状态清晰，易于理解和维护

**影响**:
- 重构了消息发送流程
- 添加了消息队列功能
- 修复了状态不一致的 bug

**替代方案**:
1. 只使用一个状态变量（被拒绝：无法区分发送中和等待回复）
2. 使用枚举状态（被拒绝：过于复杂）
3. 使用两个布尔变量（已采纳：简单清晰）

## ADR-003: 自动刷新机制

**日期**: 2026-02-08

**状态**: 已采纳

**背景**:
需要实时显示 AI 回复和工具调用，但频繁刷新会导致性能问题和界面闪烁。

**决策**:
采用双速率自动刷新策略 + 内容指纹优化。

**策略**:
- 等待 AI 回复时：2 秒间隔（快速捕获回复和工具调用）
- 空闲时：5 秒间隔（捕获后台任务回复）
- 内容指纹检查（hash），跳过无变化的重建

**理由**:
1. 平衡实时性和性能
2. 避免频繁刷新导致的闪烁
3. 减少不必要的 DOM 操作
4. 提升用户体验

**影响**:
- 添加了自动刷新定时器
- 添加了内容指纹检查
- 优化了历史记录渲染逻辑

**替代方案**:
1. 固定间隔刷新（被拒绝：无法平衡实时性和性能）
2. 只在收到事件时刷新（被拒绝：可能错过后台任务回复）
3. 双速率刷新（已采纳：最佳平衡）

## ADR-004: Agent 管理三层降级策略

**日期**: 2026-03-09

**状态**: 已采纳

**背景**:
Agent 列表获取可能因为 Gateway 未连接、CLI 未安装等原因失败，需要提供兜底方案。

**决策**:
采用三层降级策略：Gateway RPC API > CLI > 文件系统。

**策略**:
1. 优先使用 Gateway RPC API（最完整的信息）
2. 如果失败，使用 CLI 命令（`openclaw agents list --json`）
3. 如果仍失败，直接读取文件系统（`~/.openclaw/workspace/agents/`）

**理由**:
1. 提高功能可用性（即使 Gateway 未连接也能使用）
2. 兼容不同环境和配置
3. 提供最佳用户体验（优先使用最完整的信息）

**影响**:
- 添加了三层降级逻辑
- 提高了功能可用性
- 增加了代码复杂度（但可控）

**替代方案**:
1. 只使用 RPC（被拒绝：Gateway 未连接时无法使用）
2. 只使用 CLI（被拒绝：CLI 未安装时无法使用）
3. 三层降级（已采纳：最佳可用性）

## ADR-005: 创建 Agent 的角色模板设计

**日期**: 2026-03-10

**状态**: 已采纳

**背景**:
用户创建 Agent 时需要定义角色和描述，但大多数用户不知道如何写好的描述。

**决策**:
提供 6 个预设角色 + 1 个自定义角色选项，每个角色包含详细的模板描述。

**预设角色**:
1. 全栈开发 - 前后端通吃，代码质量与架构设计并重
2. 前端专家 - React/Vue/Angular，性能优化，用户体验
3. 后端专家 - API 设计，数据库优化，微服务架构
4. DevOps 工程师 - CI/CD，容器化，云原生
5. 测试工程师 - 单元测试，集成测试，自动化测试
6. 架构师 - 系统设计，技术选型，可扩展性
7. 自定义角色 - 用户自己定义

**设计**:
- 每个角色包含详细的模板描述（100-150 字）
- 用户可以在模板基础上修改
- 自定义角色使用 fullstack 作为基础模板
- 选择角色时自动填充模板到描述框

**理由**:
1. 降低用户创建 Agent 的门槛
2. 提供最佳实践参考
3. 保持灵活性（可自定义）
4. 提升用户体验

**影响**:
- 添加了角色模板数据
- 修改了创建 Agent 界面
- 优化了表单验证逻辑

**替代方案**:
1. 不提供模板（被拒绝：用户不知道如何写）
2. 只提供简单的角色名称（被拒绝：信息不够详细）
3. 提供详细的模板（已采纳：最佳用户体验）

## ADR-006: Webview 通信机制

**日期**: 2026-02-XX

**状态**: 已采纳

**背景**:
扩展代码（TypeScript）和 Webview（HTML/JS）需要双向通信。

**决策**:
使用 VSCode 的 `postMessage` API 进行通信。

**通信流程**:
- Webview → Extension: `vscode.postMessage({ type: 'xxx', ... })`
- Extension → Webview: `webview.postMessage({ type: 'xxx', ... })`

**消息类型**:
- 请求类：`sendMessage`、`refresh`、`getAgents` 等
- 响应类：`loadHistory`、`agentsList`、`connectionStatus` 等
- 事件类：`waitingReply`、`sendingComplete`、`error` 等

**理由**:
1. VSCode 官方推荐的方式
2. 简单易用，易于调试
3. 支持任意 JSON 数据

**影响**:
- 所有通信都通过 `postMessage`
- 消息类型需要明确定义
- 需要处理消息的序列化和反序列化

**替代方案**:
1. 使用 HTTP 服务器（被拒绝：过于复杂）
2. 使用 WebSocket（被拒绝：不必要）
3. 使用 postMessage（已采纳：官方推荐）

## ADR-007: 项目软连接机制

**日期**: 2026-03-09

**状态**: 已采纳

**背景**:
Agent 需要访问项目文件，但项目路径可能在任意位置。

**决策**:
在 Agent workspace 中创建项目软连接。

**位置**:
`~/.openclaw/workspace/agents/<agentId>/projects/<projectName>`

**类型**:
- Mac/Unix: symlink
- Windows: Junction

**理由**:
1. Agent 可以通过相对路径访问项目文件
2. 不需要修改项目文件
3. 支持多个项目
4. 跨平台兼容

**影响**:
- 添加了软连接创建逻辑
- 需要处理跨平台差异
- 需要处理软连接失败的情况

**替代方案**:
1. 直接使用绝对路径（被拒绝：不够灵活）
2. 复制项目文件（被拒绝：浪费空间）
3. 创建软连接（已采纳：最佳方案）

## ADR-008: 错误处理策略

**日期**: 2026-02-XX

**状态**: 已采纳

**背景**:
需要友好地处理各种错误（连接、认证、超时、余额等）。

**决策**:
分类错误类型，提供可操作的建议。

**错误分类**:
1. 连接错误（ECONNREFUSED）
2. 认证错误（401 Unauthorized）
3. 超时错误（ETIMEDOUT）
4. 余额不足（insufficient_quota）
5. 速率限制（429 Rate limit）
6. 上下文超限（context_length）
7. 模型不可用（model_not_found）
8. 其他错误

**处理方式**:
- 解析错误信息
- 分类错误类型
- 显示友好的错误提示
- 提供可操作的建议（如何解决）

**理由**:
1. 提升用户体验
2. 减少用户困惑
3. 提供解决方案
4. 避免技术术语

**影响**:
- 添加了错误解析逻辑
- 添加了友好的错误提示
- 提高了用户满意度

**替代方案**:
1. 直接显示原始错误（被拒绝：用户看不懂）
2. 只显示简单提示（被拒绝：信息不够）
3. 分类处理（已采纳：最佳用户体验）

## ADR-009: 多窗口支持

**日期**: 2026-02-XX

**状态**: 已采纳

**背景**:
用户可能需要同时进行多个对话（如同时问不同问题）。

**决策**:
支持最多 5 个并行聊天会话（独立面板）。

**实现**:
- 每个窗口有独立的 `ChatPanel` 实例
- 每个窗口有独立的 `sessionKey`
- 每个窗口有独立的历史记录
- 窗口 ID 格式：`vscode-<timestamp>`

**限制**:
- 最多 5 个并行会话
- 达到上限时显示友好提示

**理由**:
1. 提高工作效率（并行处理多个任务）
2. 避免资源占用过高（限制数量）
3. 提供良好的用户体验

**影响**:
- 添加了窗口管理逻辑
- 添加了窗口数量限制
- 增加了内存占用（可控）

**替代方案**:
1. 只支持单窗口（被拒绝：不够灵活）
2. 无限制窗口（被拒绝：资源占用过高）
3. 限制 5 个窗口（已采纳：平衡性能和体验）

## ADR-010: 国际化支持

**日期**: 2026-02-XX

**状态**: 已采纳

**背景**:
需要支持多语言用户（中文、英文、日语、韩语）。

**决策**:
实现完整的 i18n 支持，包括 UI 和 AI 输出。

**实现**:
- UI 语言：通过 `package.nls.json` 和 `package.nls.zh-cn.json`
- AI 输出语言：通过 `openclaw.aiOutputLanguage` 配置
- Webview 语言：通过 `setLocale` 消息同步

**支持语言**:
- 中文（zh-CN）
- 英文（en）
- 日语（ja）
- 韩语（ko）

**理由**:
1. 扩大用户群体
2. 提升用户体验
3. 符合国际化标准

**影响**:
- 添加了多语言资源文件
- 添加了语言切换逻辑
- 增加了维护成本（需要翻译）

**替代方案**:
1. 只支持英文（被拒绝：限制用户群体）
2. 只支持中文（被拒绝：限制用户群体）
3. 支持多语言（已采纳：最佳国际化）
