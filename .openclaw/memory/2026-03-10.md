## 详细过程

### 修复"正在思考中"提示消失问题

**问题分析**：
1. 用户发送消息时，`sendMessageNow` 会调用 `showThinking()` 显示提示
2. 后端收到消息后，发送 `waitingReply` 消息，前端设置 `chatRunId`
3. 启动自动刷新，每 2 秒刷新一次历史记录
4. `loadHistory` 会清空所有消息（`messages.innerHTML = ''`）
5. 虽然有逻辑在重建后检查 `isBusy()` 并重新显示 thinking indicator，但在 `waitingReply` 消息处理中没有调用 `showThinking()`

**解决方案**：
在 `waitingReply` 消息处理中添加 `showThinking()` 调用：

```javascript
case 'waitingReply':
    // RPC 已发送，等待 AI 回复（chatRunId 追踪状态）
    isSending = false;
    chatRunId = message.runId || true;
    updateSendButtonState();
    showThinking(); // 确保显示"正在思考中"提示
    startAutoRefresh(autoRefreshInterval);
    break;
```

**测试结果**：
- ✅ 发送消息后，"正在思考中"提示正常显示
- ✅ 自动刷新时，提示不会消失
- ✅ AI 回复完成后，提示正常隐藏

## 遇到的问题

### 1. "正在思考中"提示消失的根本原因
**问题**：一开始不确定是什么原因导致提示消失。

**分析过程**：
1. 检查 `showThinking()` 和 `hideThinking()` 的调用位置
2. 发现 `loadHistory` 会清空所有消息
3. 发现 `loadHistory` 后有逻辑重新显示 thinking indicator
4. 但是 `waitingReply` 处理中没有调用 `showThinking()`
5. 可能在 `waitingReply` 和第一次 `loadHistory` 之间，thinking indicator 被清除了

**解决**：在 `waitingReply` 处理中添加 `showThinking()` 调用，确保始终显示。

### 2. 自定义角色的处理方式
**问题**：自定义角色应该使用什么作为基础模板？

**考虑方案**：
1. 不使用模板，让用户完全自己写（太难）
2. 使用一个通用模板（不够灵活）
3. 使用 fullstack 作为基础模板（最合适）

**决策**：使用 fullstack 作为基础模板，并在描述中添加自定义角色信息。

### 3. 表单验证的完整性
**问题**：需要确保所有必填字段都已填写。

**解决**：
- ID、名称、Emoji、角色、描述都是必填
- 如果选择自定义角色，还需要填写自定义角色名称
- 实时更新创建按钮的 disabled 状态

## 经验教训

### 1. 状态管理要全面
不仅要在初始化时设置状态，还要在状态变化时及时更新。`waitingReply` 时应该确保 thinking indicator 显示，而不是依赖之前的状态。

### 2. 用户体验要细致
创建 Agent 时，提供预设模板可以大大降低用户的使用门槛，同时保持灵活性（可自定义）。

### 3. 文档要及时更新
代码变更后要及时更新文档，否则文档会过时，对新 Agent 或新开发者不友好。

### 4. 测试要全面
修复 bug 后要测试各种场景，确保没有引入新的问题。

### 5. 代码要易于维护
使用清晰的命名、合理的结构、充分的注释，让代码易于理解和维护。
